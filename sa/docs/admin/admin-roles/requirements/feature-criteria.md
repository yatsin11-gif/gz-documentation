# Критерии

## Источник

- Feature spec: `sa/docs/admin/admin-roles/requirements/feature-spec.md`
- Утвердил: Не утверждено; draft подготовлен по запросу SA
- Дата: 2026-06-30
- Репозитории в скоупе: `gozap-admin`, `gozap-backend`

## Цель

- Супер-админ должен иметь возможность создавать администраторов и назначать им набор конкретных разделов (множественный выбор из списка: «Станции», «Пользователи», «Партнёры»).
- Супер-админ должен иметь возможность создавать курьеров в той же форме, что и администраторов (поля: логин, пароль, статус; статус по умолчанию «активен»).
- Администратор должен видеть в админке только те разделы, которые ему были назначены при создании.

## FE Criteria

### Затронутые поверхности

- `gozap-admin`: раздел управления администраторами и курьерами (доступен только супер-админу), навигация админки с условной видимостью разделов.

### Пользовательские сценарии и состояния

#### Создание администратора (супер-админ)

- Супер-админ должен иметь возможность создать нового администратора.
- При создании администратора супер-админ должен отметить множественным выбором один или несколько разделов из списка: «Станции», «Пользователи», «Партнёры».
- Форма создания администратора не должна позволять сохранение без хотя бы одного отмеченного раздела.

#### Создание курьера (супер-админ)

- Супер-админ должен иметь возможность создать нового курьера в той же форме, что и администратора.
- Поля при создании курьера: логин, пароль, статус (по умолчанию «активен»).
- Блокировка курьера необратима — разблокировка невозможна.

#### Вход и навигация (администратор)

- Администратор должен иметь возможность войти в админку.
- После входа администратор должен видеть только те разделы из списка («Станции», «Пользователи», «Партнёры»), которые были ему назначены при создании.
- Администратор не должен видеть в навигации пункты меню для неназначенных разделов: пункты меню недоступных разделов должны полностью отсутствовать в боковом меню, а не отображаться заблокированными/неактивными.

### Правила пользовательского поведения

- Супер-админ видит и может управлять всеми разделами без ограничений.
- Администратор не должен иметь возможность изменить собственный набор доступных разделов.
- Супер-админ не должен иметь возможность отредактировать данные уже созданного администратора (логин, имя и т. п.) или изменить его набор назначенных разделов — UI не должен предоставлять форму/действие редактирования администратора после создания.

### Ограничения и исключения на FE

- Критерии не фиксируют отдельные права внутри раздела (например, отдельно «создание» и «архивирование» станции как разные права) — доступ выдаётся на раздел целиком.
- Критерии не требуют интерфейса аудит-лога действий администраторов.

## BE Criteria

### Затронутые backend области

- `gozap-backend`: учётные записи администраторов и курьеров, роли (супер-админ, админ, курьер) и набор назначенных разделов, авторизация запросов по разделам.
- Внешние интеграции: N/A.

### Правила бизнес-логики и обработки

#### Создание администратора

- Backend должен поддерживать создание учётной записи администратора только от имени супер-админа.
- Backend должен сохранять набор разделов (один или несколько из: «Станции», «Пользователи», «Партнёры»), выбранный при создании администратора.
- Backend должен отклонять создание администратора без хотя бы одного указанного раздела.

#### Создание курьера

- Backend должен поддерживать создание учётной записи курьера только от имени супер-админа.
- Поля курьера: логин, пароль, статус (по умолчанию «активен»).
- Backend должен поддерживать блокировку курьера; разблокировка невозможна (блокировка необратима).

#### Авторизация по разделам

- Backend должен проверять, входит ли запрашиваемый раздел/эндпоинт в набор разделов, назначенных администратору.
- Backend не должен возвращать данные разделов, не входящих в назначенный администратору набор.
- Backend должен разрешать супер-админу доступ ко всем разделам без дополнительной проверки набора.

#### Редактирование администратора

- Backend не должен предоставлять эндпоинт для редактирования данных уже созданного администратора или его набора назначенных разделов — данные и доступ фиксируются на этапе создания и далее неизменны.

### Побочные эффекты и интеграционные результаты

- Платежи: N/A.
- Подписка / доступ: доступ администратора к разделам определяется набором разделов, заданным супер-админом при создании.
- Аналитика: Не определено.
- Email / CRM / webhooks: Не определено.

### Ограничения и исключения на BE

- Критерии не фиксируют точную схему хранения ролей/permissions.
- Критерии не требуют поддержки делегирования роли супер-админа.
- Критерии этой фичи описывают только модель доступа (кто видит какой раздел); CRUD-логика внутри разделов «Станции» и «Пользователи» специфицирована отдельно (`sa/docs/admin/stations-management`, `sa/docs/admin/user-management`), раздел «Партнёры» — в `sa/docs/partners/partner-program`; детальная функциональность роли курьера — в `sa/docs/admin/courier-management`.

## Нефункциональные критерии

### FE

- Производительность: Не определено.
- Доступность: разделы, не назначенные администратору, не должны быть доступны прямым переходом по URL.
- Консистентность UX: единый паттерн отображения/скрытия разделов навигации по назначенному набору.

### BE

- Надежность: Не определено.
- Наблюдаемость: backend должен различать ошибки аутентификации (неверный логин/пароль) и ошибки авторизации (раздел не входит в назначенный набор).
- Безопасность: проверка набора разделов должна выполняться на backend, а не только скрытием элементов на FE.

## Вне scope

- Гранулярные права на уровне отдельных действий внутри раздела.
- Аудит-лог действий администраторов.
- Делегирование/передача роли супер-админа.
- Редактирование данных и набора разделов уже созданного администратора (подтверждено SA — функциональность не предусмотрена).
- Создание или изменение Qase test cases.

## Открытые вопросы

- Какой именно ответ должен возвращать backend/FE при прямом обращении администратора по ссылке к недоступному разделу — ошибка 403 или редирект (скрытие пункта в боковом меню уже подтверждено)?

## Примечания

- Критерии подготовлены на основе draft feature spec и уточнения SA по списку разделов («Станции», «Пользователи», «Партнёры») и их функциональному составу.
- FE отвечает за форму создания администратора и курьера (общая форма, множественный выбор разделов для админа) и условную навигацию по назначенному набору.
- Backend отвечает за хранение набора разделов на администратора, хранение учётных записей курьеров и серверную проверку доступа к разделам.
- Роли: супер-админ (полный доступ), админ (набор разделов), курьер (фиксированный набор, см. `sa/docs/admin/courier-management`); партнёр — отдельная ролевая ось.
