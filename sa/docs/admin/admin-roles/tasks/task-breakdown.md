## Список задач

### Task 1

- Title: BE. Admin Roles. Модель ролей/уровней доступа и серверная авторизация
- Repository: `gozap-backend`
- Owner role: Backend engineer
- Зачем нужна задача: без серверной модели ролей и проверки доступа нельзя гарантировать, что администратор не получит данные недоступных разделов даже при обходе FE.
- Scope:
  - модель ролей: супер-админ (полный доступ), админ (набор назначенных разделов), курьер (фиксированный набор разрешённых разделов/действий)
  - набор разделов для назначения администратору: «Станции», «Пользователи», «Партнёры»
  - создание администратора только от имени супер-админа с обязательным множественным выбором хотя бы одного раздела
  - создание курьера только от имени супер-админа (поля: логин, пароль, статус; статус по умолчанию «активен»); блокировка курьера необратима
  - серверная проверка набора разделов при обращении к ограниченным разделам/эндпоинтам
- Out of scope:
  - гранулярные права на уровне отдельных действий внутри раздела
  - аудит-лог действий администраторов
  - CRUD-логика внутри разделов «Станции»/«Пользователи»/«Партнёры» (отдельные фичи)
  - редактирование данных и набора разделов уже созданного администратора (подтверждено SA — функциональность не предусмотрена, эндпоинт редактирования не нужен)
- Dependencies:
  - нет
- Risks:
  - нет
- Definition of done:
  - супер-админ может создать администратора с набором разделов через API
  - супер-админ может создать курьера через API (логин, пароль, статус)
  - запросы администратора к неназначенным разделам отклоняются на backend

### Task 2

- Title: FE. Admin Roles. Создание администратора и условная навигация
- Repository: `gozap-admin`
- Owner role: Frontend engineer
- Зачем нужна задача: супер-админу нужен интерфейс создания администраторов, а администратору — навигация, ограниченная его уровнем доступа.
- Scope:
  - форма создания администратора с множественным выбором разделов («Станции», «Пользователи», «Партнёры»), обязателен хотя бы один (доступна только супер-админу)
  - та же форма используется для создания курьера (поля: логин, пароль, статус; статус по умолчанию «активен»)
  - условное отображение пунктов навигации по набору разделов текущего пользователя: пункты меню недоступных разделов полностью отсутствуют (не отображаются заблокированными)
- Out of scope:
  - редактирование данных и набора разделов уже созданного администратора (подтверждено SA — функциональность не предусмотрена)
- Dependencies:
  - Task 1
- Risks:
  - нет
- Definition of done:
  - супер-админ может создать администратора с набором разделов
  - супер-админ может создать курьера через ту же форму (логин, пароль, статус)
  - администратор видит в навигации только назначенные ему разделы; пункты недоступных разделов в боковом меню отсутствуют

## Порядок поставки

1. Task 1 — серверная модель ролей и авторизация.
2. Task 2 — FE создание администратора и условная навигация.

## Cross-Repository Notes

- Ролевая модель (супер-админ/админ/курьер) — отдельная ролевая ось от роли `partner` из `sa/docs/partners/partner-program`.
- Разделы «Станции» и «Пользователи», доступ к которым регулирует эта фича, специфицированы отдельными фичами `sa/docs/admin/stations-management` и `sa/docs/admin/user-management`.
- Детальная функциональность роли курьера — в `sa/docs/admin/courier-management`; эта фича специфицирует включение курьера в ролевую модель и создание через общую форму.
