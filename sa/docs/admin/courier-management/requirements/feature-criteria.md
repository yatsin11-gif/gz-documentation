# Критерии

## Источник

- Feature spec: `sa/docs/admin/courier-management/requirements/feature-spec.md`
- Утвердил: Не утверждено; draft подготовлен по запросу SA
- Дата: 2026-07-02
- Репозитории в скоупе: `gozap-admin`, `gozap-backend`

## Цель

- Курьер должен иметь возможность войти в систему, видеть список станций с количеством ПБ и свободных слотов, открывать карточку станции (только чтение) и выполнять операции с ПБ (выдача без QR, выдача всех, возврат).
- Супер-админ должен управлять курьерами: видеть список, редактировать данные и статус, блокировать.

## FE Criteria

### Затронутые поверхности

- `gozap-admin`: интерфейс курьера — список станций, карточка станции (только чтение), операции с ПБ.
- `gozap-admin`: раздел управления курьерами для супер-админа — список курьеров, редактирование, блокировка.

### Пользовательские сценарии и состояния

#### Вход курьера

- Курьер входит в систему под своей учётной записью и видит только доступные разделы и действия.
- Срок действия доступа не ограничен тайм-лимитом.
- Заблокированный курьер при попытке входа должен получать ошибку.

#### Список станций (курьер)

- Курьер должен видеть список всех станций с полями: код станции, адрес, количество ПБ, количество свободных слотов, статус станции.
- Курьер должен иметь возможность сортировать список по количеству ПБ по убыванию и по возрастанию.

#### Карточка станции (курьер)

- Курьер видит в карточке станции: адрес станции, код станции, заводской номер станции, фото станции, график работы.
- Курьер не может редактировать или архивировать станцию.
- В карточке отображаются только разрешённые курьеру действия.

#### Операции с ПБ (курьер)

- Курьер может достать ПБ без QR-кода.
- Курьер может нажать кнопку «выдать все ПБ».
- Курьер может вернуть ненужные ПБ.
- Если станция в офлайне, операции обслуживания недоступны.

#### Создание курьера (супер-админ)

- Супер-админ создаёт курьера в той же форме, что и остальных администраторов (см. `sa/docs/admin/admin-roles`).
- При создании задаются: логин, пароль; статус по умолчанию «активен».

#### Список курьеров (супер-админ)

- Супер-админ должен видеть список всех курьеров с полями: имя, статус, контакты, действия.

#### Редактирование курьера (супер-админ)

- Супер-админ должен иметь возможность редактировать данные курьера.
- Супер-админ должен иметь возможность менять статус курьера.

#### Блокировка курьера (супер-админ)

- Супер-админ должен иметь возможность заблокировать курьера.
- Заблокированный курьер при авторизации получает ошибку.
- Разблокировка невозможна: блокировка необратима.

### Правила пользовательского поведения

- Курьер видит только разрешённые ему разделы и действия, без доступа к административным функциям.
- Управление курьерами доступно только супер-админу.

### Ограничения и исключения на FE

- Критерии не описывают редактирование или архивирование станции курьером.
- Критерии не описывают привязку курьера к конкретным станциям.
- Разблокировка курьера невозможна (подтверждено SA).

## BE Criteria

### Затронутые backend области

- `gozap-backend`: сущность «курьер» (учётная запись с ролью курьера), авторизация по роли, API списка станций для курьера, API операций с ПБ, API управления курьерами для супер-админа.
- Внешние интеграции: N/A.

### Правила бизнес-логики и обработки

#### Учётная запись курьера

- Учётная запись курьера содержит поля: логин, пароль, статус. Статус по умолчанию — «активен».
- Backend должен поддерживать создание учётной записи курьера супер-админом; форма создания та же, что для создания администраторов (см. `sa/docs/admin/admin-roles`).
- Backend должен поддерживать редактирование данных и статуса курьера супер-админом.
- Backend должен поддерживать блокировку курьера: заблокированный курьер не должен проходить авторизацию.
- Разблокировка курьера невозможна: блокировка необратима.

#### Список станций для курьера

- Backend должен возвращать список всех станций с полями: код станции, адрес, количество ПБ, количество свободных слотов, статус станции.
- Backend должен поддерживать сортировку по количеству ПБ по убыванию и по возрастанию.

#### Карточка станции для курьера

- Backend должен возвращать данные карточки станции: адрес станции, код станции, заводской номер станции, фото станции, график работы (только чтение).

#### Операции с ПБ

- Backend должен поддерживать выдачу ПБ без QR-кода по запросу курьера.
- Backend должен поддерживать выдачу всех ПБ из станции по запросу курьера.
- Backend должен поддерживать возврат ПБ в станцию по запросу курьера.
- Backend должен отклонять операции с ПБ на станции в офлайне.

#### Авторизация

- Backend должен проверять, что запросы курьера ограничены его ролью (без доступа к административным функциям).
- Backend должен проверять, что управление курьерами выполняется только супер-админом.

### Побочные эффекты и интеграционные результаты

- Платежи: N/A.
- Подписка / доступ: роль курьера интегрируется в ролевую модель `gozap-admin`.
- Аналитика: Не определено.
- Email / CRM / webhooks: Не определено.

### Ограничения и исключения на BE

- Разблокировка курьера невозможна (подтверждено SA).
- Критерии не описывают привязку курьера к конкретным станциям.
- Критерии не описывают логирование операций курьера с ПБ.

## Нефункциональные критерии

### FE

- Производительность: Не определено.
- Доступность: разделы курьера не должны быть доступны пользователям с другими ролями (администратор, партнёр).
- Консистентность UX: список станций должен явно отображать количество ПБ, свободных слотов и статус станции; статус блокировки курьера должен быть виден в списке.

### BE

- Надежность: Не определено.
- Наблюдаемость: Не определено.
- Безопасность: проверка роли курьера должна выполняться на backend, а не только скрытием элементов на FE.

## Вне scope

- Редактирование и архивирование станции курьером.
- Привязка курьера к конкретным станциям.
- Разблокировка курьера.
- Тайм-лимит на доступ курьера.
- Создание или изменение Qase test cases.

## Открытые вопросы

- Нужно ли логирование операций курьера с ПБ?
- Нужна ли курьеру возможность сброса пароля?

## Примечания

- Критерии подготовлены на основе draft feature spec по входным требованиям «Логика работы роли "Курьер"».
- FE отвечает за интерфейс курьера (станции, ПБ-операции) и за интерфейс супер-админа по управлению курьерами.
- Backend отвечает за сущность курьера, авторизацию по роли, API станций для курьера, API операций с ПБ и API управления курьерами.
