# Критерии

## Источник

- Feature spec: `sa/docs/admin/courier-management/requirements/feature-spec.md`
- Утвердил: Не утверждено; draft подготовлен по запросу SA
- Дата: 2026-07-02
- Репозитории в скоупе: `gozap-admin`, `gozap-backend`

## Цель

- Курьер должен иметь возможность войти в систему, видеть список назначенных ему станций с количеством ПБ и свободных слотов, открывать карточку станции (только чтение), выполнять операции с ПБ (забор ПБ по количеству, добавление ПБ через «начать»/«завершить») и сообщать о проблеме по станции через комментарий.
- Супер-админ должен управлять курьерами: видеть список, редактировать данные и статус, блокировать, назначать/удалять станции курьеру и просматривать журнал действий.

## FE Criteria

### Затронутые поверхности

- `gozap-admin`: интерфейс курьера — список назначенных станций, карточка станции (только чтение), операции с ПБ, отправка комментария о проблеме.
- `gozap-admin`: раздел управления курьерами для супер-админа — список курьеров, редактирование, блокировка, назначение/удаление станций, журнал действий.

### Пользовательские сценарии и состояния

#### Вход курьера

- Курьер входит в систему под своей учётной записью и видит только доступные разделы и действия.
- Срок действия доступа не ограничен тайм-лимитом.
- Заблокированный курьер при попытке входа должен получать ошибку.

#### Список станций (курьер)

- Курьер должен видеть список назначенных ему станций с полями: код станции, адрес, количество ПБ, количество свободных слотов, статус станции.
- Курьер должен иметь возможность сортировать список по количеству ПБ по убыванию и по возрастанию.
- Курьер не должен видеть станции, которые ему не назначены.

#### Карточка станции (курьер)

- Курьер видит в карточке станции: адрес станции, код станции, заводской номер станции, фото станции, график работы.
- Курьер не может редактировать или архивировать станцию.
- В карточке отображаются только разрешённые курьеру действия.

#### Операции с ПБ (курьер)

- Операции возможны только со станцией, к которой курьер прикреплён (назначена), и только если станция не в офлайне.
- Забор ПБ: по нажатию кнопки выдачи курьер вводит количество ПБ; минимум 1, максимум — все доступные на станции на момент запроса; выдача без QR-кода.
- Забор ПБ завершается статусом «успех», если запрошенное количество выдано; статусом «ошибка», если станция ушла в офлайн в процессе.
- Добавление ПБ: курьер нажимает «начать добавление пауэрбанков» (фиксируется время начала), затем «завершить добавление пауэрбанков» (фиксируется время завершения).
- Кнопка «завершить добавление пауэрбанков» доступна только если добавление было начато этим курьером на этой станции.
- При ошибке операции (забор или добавление) курьер может добавить комментарий к действию.
- Каждая операция получает верхнеуровневый статус: «успех», «ошибка», «отмена», «отказ».
- Если станция в офлайне или данные не синхронизированы, операция получает понятный статус, не теряется, но не может завершиться успехом.

#### Сообщение о проблеме по станции (курьер)

- Курьер должен иметь возможность сообщить о проблеме по станции (станция повреждена, не работает, не совпадает количество ПБ) через комментарий.
- Комментарий о проблеме должен сохраняться и быть доступен супер-админу (в журнале действий).

#### Создание курьера (супер-админ)

- Супер-админ создаёт курьера в той же форме, что и остальных администраторов (см. `sa/docs/admin/admin-roles`).
- При создании задаются: логин, пароль; статус по умолчанию «активен».

#### Список курьеров (супер-админ)

- Супер-админ должен видеть список всех курьеров с полями: имя, статус, контакты, действия.

#### Редактирование курьера (супер-админ)

- Супер-админ должен иметь возможность редактировать данные курьера.
- Супер-админ должен иметь возможность менять статус курьера.

#### Блокировка курьера (супер-админ)

- Супер-админ должен иметь возможность заблокировать курьера.
- Заблокированный курьер при авторизации получает ошибку.
- Разблокировка невозможна: блокировка необратима.

#### Назначение станций курьеру (супер-админ)

- Супер-админ должен иметь возможность вручную назначить курьеру список станций.
- Супер-админ должен видеть назначенные станции в карточке курьера.
- Супер-админ должен иметь возможность удалить назначение станции курьеру.

#### Журнал действий (супер-админ)

- Супер-админ должен видеть журнал действий курьеров по операциям с ПБ.
- В журнале по каждой записи отображается: логин курьера, id станции, совершённое действие (забор/добавление ПБ), дата/время действия (для добавления — время завершения), статус станции, количество ПБ до операции, количество ПБ после операции, статус действия, комментарий курьера (при наличии).

### Правила пользовательского поведения

- Курьер видит только разрешённые ему разделы и действия, без доступа к административным функциям.
- Курьер видит только назначенные ему станции.
- Управление курьерами (включая назначение станций и журнал действий) доступно только супер-админу.

### Ограничения и исключения на FE

- Критерии не описывают редактирование или архивирование станции курьером.
- Разблокировка курьера невозможна (подтверждено SA).

## BE Criteria

### Затронутые backend области

- `gozap-backend`: сущность «курьер» (учётная запись с ролью курьера), назначение станций курьеру, авторизация по роли, API списка станций для курьера, API операций с ПБ (со статусами), API комментария о проблеме, журнал действий, API управления курьерами для супер-админа.
- Внешние интеграции: N/A.

### Правила бизнес-логики и обработки

#### Учётная запись курьера

- Учётная запись курьера содержит поля: логин, пароль, статус. Статус по умолчанию — «активен».
- Backend должен поддерживать создание учётной записи курьера супер-админом; форма создания та же, что для создания администраторов (см. `sa/docs/admin/admin-roles`).
- Backend должен поддерживать редактирование данных и статуса курьера супер-админом.
- Backend должен поддерживать блокировку курьера: заблокированный курьер не должен проходить авторизацию.
- Разблокировка курьера невозможна: блокировка необратима.

#### Назначение станций курьеру

- Backend должен поддерживать назначение супер-админом списка станций конкретному курьеру.
- Backend должен возвращать список назначенных станций в карточке курьера.
- Backend должен поддерживать удаление назначения станции курьеру.

#### Список станций для курьера

- Backend должен возвращать список станций, назначенных курьеру, с полями: код станции, адрес, количество ПБ, количество свободных слотов, статус станции.
- Backend не должен возвращать курьеру станции, которые ему не назначены.
- Backend должен поддерживать сортировку по количеству ПБ по убыванию и по возрастанию.

#### Карточка станции для курьера

- Backend должен возвращать данные карточки станции: адрес станции, код станции, заводской номер станции, фото станции, график работы (только чтение).

#### Операции с ПБ

- Backend должен разрешать операции только со станцией, к которой прикреплён курьер, и только если станция не в офлайне.
- Забор ПБ: backend должен принимать количество ПБ для выдачи; минимум 1, максимум — все доступные на станции на момент запроса; выдача без QR-кода.
- Backend должен возвращать статус «успех», если запрошенное количество ПБ выдано, и «ошибка», если станция ушла в офлайн в процессе выдачи.
- Добавление ПБ: backend должен фиксировать время начала («начать добавление») и время завершения («завершить добавление»); завершение возможно только если добавление было начато этим курьером на этой станции.
- Backend должен принимать комментарий курьера к действию при ошибке (забор или добавление).
- Каждая операция должна получать верхнеуровневый статус: «успех», «ошибка», «отмена», «отказ».
- Backend должен отклонять операции с ПБ на станции в офлайне или при несинхронизированных данных: операция получает понятный статус, не теряется, но не завершается успехом.

#### Сообщение о проблеме по станции

- Backend должен принимать и сохранять комментарий курьера о проблеме по станции (повреждена, не работает, не совпадает количество ПБ).
- Комментарий должен быть связан с операцией/станцией и доступен супер-админу в журнале действий.

#### Журнал действий

- Backend должен фиксировать каждую операцию курьера с ПБ в журнале действий.
- Запись журнала должна содержать: логин курьера, id станции, действие (забор/добавление ПБ), дата/время действия (для добавления — время завершения), статус станции, количество ПБ до операции, количество ПБ после операции, статус действия, комментарий курьера (при наличии).
- Журнал действий должен быть доступен только супер-админу.

#### Авторизация

- Backend должен проверять, что запросы курьера ограничены его ролью (без доступа к административным функциям) и только к назначенным ему станциям.
- Backend должен проверять, что управление курьерами (включая назначение станций и журнал действий) выполняется только супер-админом.

### Побочные эффекты и интеграционные результаты

- Платежи: N/A.
- Подписка / доступ: роль курьера интегрируется в ролевую модель `gozap-admin`.
- Аналитика: Не определено.
- Email / CRM / webhooks: Не определено.

### Ограничения и исключения на BE

- Разблокировка курьера невозможна (подтверждено SA).

## Нефункциональные критерии

### FE

- Производительность: Не определено.
- Доступность: разделы курьера не должны быть доступны пользователям с другими ролями (администратор, партнёр).
- Консистентность UX: список станций должен явно отображать количество ПБ, свободных слотов и статус станции; статус блокировки курьера должен быть виден в списке; статус каждой операции должен быть понятен пользователю.

### BE

- Надежность: Не определено.
- Наблюдаемость: Не определено.
- Безопасность: проверка роли курьера должна выполняться на backend, а не только скрытием элементов на FE.

## Вне scope

- Редактирование и архивирование станции курьером.
- Разблокировка курьера.
- Тайм-лимит на доступ курьера.
- Создание или изменение Qase test cases.

## Примечания

- Критерии подготовлены на основе draft feature spec по входным требованиям «Логика работы роли "Курьер"».
- FE отвечает за интерфейс курьера (назначенные станции, ПБ-операции, комментарий о проблеме) и за интерфейс супер-админа по управлению курьерами (включая назначение станций и журнал действий).
- Backend отвечает за сущность курьера, назначение станций, авторизацию по роли, API станций для курьера, API операций с ПБ (со статусами), комментарий о проблеме, журнал действий и API управления курьерами.
