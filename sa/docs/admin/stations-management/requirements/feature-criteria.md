# Критерии

## Источник

- Feature spec: `sa/docs/admin/stations-management/requirements/feature-spec.md`
- Утвердил: Не утверждено; draft подготовлен по запросу SA
- Дата: 2026-06-30
- Репозитории в скоупе: `gozap-admin`, `gozap-backend`

## Цель

- Администратор с доступом к разделу «Станции» должен видеть список станций, создавать новую станцию и архивировать существующую.

## FE Criteria

### Затронутые поверхности

- `gozap-admin`: раздел «Станции» — список станций, форма создания станции, действие архивирования.

### Пользовательские сценарии и состояния

#### Список станций

- Администратор должен видеть список всех станций с указанием статуса (активна/архивирована).

#### Создание станции

- Администратор должен иметь возможность открыть форму создания новой станции и сохранить её.
- Новая станция должна появляться в списке станций сразу после создания.

#### Архивирование станции

- Администратор должен иметь возможность архивировать станцию из списка или карточки станции.
- Архивированная станция должна оставаться в списке с явным статусом «архивирована».

### Правила пользовательского поведения

- Раздел «Станции» должен быть доступен только администратору с назначенным разделом «Станции» или супер-админу.
- Архивированную станцию нельзя архивировать повторно (действие должно быть недоступно/неактивно).

### Ограничения и исключения на FE

- Критерии не фиксируют точный набор полей формы создания станции (см. открытые вопросы).
- Критерии не описывают разархивирование станции.
- Критерии не описывают редактирование уже созданной станции.

## BE Criteria

### Затронутые backend области

- `gozap-backend`: сущность «станция» (создание, статус активна/архивирована), список станций, авторизация доступа к разделу «Станции» по ролевой модели `sa/docs/admin/admin-roles`.
- Внешние интеграции: N/A.

### Правила бизнес-логики и обработки

#### Создание станции

- Backend должен создавать сущность «станция» по запросу администратора с доступом к разделу «Станции».
- Backend должен присваивать новой станции статус «активна» по умолчанию.

#### Архивирование станции

- Backend должен переводить станцию в статус «архивирована» по запросу администратора с доступом к разделу «Станции».
- Backend не должен удалять исторические данные станции (привязки к партнёрам, аренды) при архивировании.

#### Авторизация

- Backend должен проверять, что запрос на создание/архивирование станции выполняется администратором, которому назначен раздел «Станции», или супер-админом.

### Побочные эффекты и интеграционные результаты

- Платежи: N/A.
- Подписка / доступ: доступ к операциям регулируется набором разделов администратора (`sa/docs/admin/admin-roles`).
- Аналитика: Не определено.
- Email / CRM / webhooks: Не определено.

### Ограничения и исключения на BE

- Критерии не фиксируют точную схему полей станции.
- Критерии не описывают поведение архивирования станции с активными арендами или активной привязкой к партнёру (см. открытые вопросы).

## Нефункциональные критерии

### FE

- Производительность: Не определено.
- Доступность: раздел «Станции» не должен быть доступен прямым переходом по URL администратору без назначенного раздела.
- Консистентность UX: список станций должен визуально отличать активные и архивированные станции.

### BE

- Надежность: Не определено.
- Наблюдаемость: Не определено.
- Безопасность: проверка доступа к разделу «Станции» должна выполняться на backend, а не только скрытием элементов на FE.

## Вне scope

- Редактирование уже созданной станции.
- Удаление станции (только архивирование).
- Привязка станции к партнёру и процент вознаграждения (`sa/docs/partners/partner-program`).
- Разархивирование станции.
- Создание или изменение Qase test cases.

## Открытые вопросы

- Какой набор полей должен быть у станции при создании?
- Что должно происходить при попытке архивировать станцию, привязанную к партнёру с активными арендами?
- Нужно ли разархивирование станции в будущей итерации?

## Примечания

- Критерии подготовлены на основе draft feature spec и уточнения SA по составу раздела «Станции».
- FE отвечает за список станций, форму создания и действие архивирования.
- Backend отвечает за хранение сущности станции, переходы статуса и проверку доступа по разделу.
