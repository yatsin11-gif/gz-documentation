# Критерии

## Источник

- Feature spec: `sa/docs/admin/user-management/requirements/feature-spec.md`
- Утвердил: Не утверждено; draft подготовлен по запросу SA
- Дата: 2026-06-30
- Репозитории в скоупе: `gozap-admin`, `gozap-backend`

## Цель

- Администратор с доступом к разделу «Пользователи» должен видеть список пользователей, создавать пользователя, выдавать/снимать промоаккаунт, блокировать пользователя и просматривать карточку пользователя.

## FE Criteria

### Затронутые поверхности

- `gozap-admin`: раздел «Пользователи» — список пользователей, форма создания пользователя, карточка пользователя с действиями (промоаккаунт, блокировка).

### Пользовательские сценарии и состояния

#### Список пользователей

- Администратор должен видеть список всех пользователей независимо от способа создания (вручную администратором или автоматически через станцию, см. `sa/docs/kiosk/terminal-rental-flow`).

#### Создание пользователя

- Администратор должен иметь возможность открыть форму создания нового пользователя и сохранить её.
- Новый пользователь должен появляться в списке сразу после создания.

#### Карточка пользователя

- Карточка должна отображать: имя, номер телефона, почту, данные о текущей аренде (если есть активная аренда), историю аренд пользователя.
- Карточка должна быть доступна как для пользователей, созданных вручную, так и для созданных через станцию.

#### Промоаккаунт

- Администратор должен иметь возможность выдать промоаккаунт пользователю из карточки пользователя.
- Администратор должен иметь возможность снять промоаккаунт у пользователя из карточки пользователя.
- Статус промоаккаунта должен быть виден в карточке пользователя.

#### Блокировка

- Администратор должен иметь возможность заблокировать пользователя из карточки пользователя.
- Статус блокировки должен быть виден в карточке пользователя и/или в списке пользователей.

### Правила пользовательского поведения

- Раздел «Пользователи» должен быть доступен только администратору с назначенным разделом «Пользователи» или супер-админу.
- Промоаккаунт и блокировка — независимые статусы; выдача/снятие одного не должно влиять на другой.

### Ограничения и исключения на FE

- Критерии не описывают редактирование данных пользователя (имя, телефон, почта) после создания.
- Критерии не описывают удаление пользователя.
- Критерии не описывают разблокировку пользователя (см. открытые вопросы).
- Критерии не фиксируют обязательные поля формы создания пользователя, кроме телефона по аналогии со станцией (см. открытые вопросы).

## BE Criteria

### Затронутые backend области

- `gozap-backend`: сущность «пользователь» (та же, что создаётся автоматически через станцию, см. `sa/docs/kiosk/terminal-rental-flow`), статусы промоаккаунта и блокировки, история аренд, авторизация доступа к разделу «Пользователи» по ролевой модели `sa/docs/admin/admin-roles`.
- Внешние интеграции: N/A.

### Правила бизнес-логики и обработки

#### Создание пользователя

- Backend должен создавать пользователя по запросу администратора с доступом к разделу «Пользователи».
- Если пользователь с указанным номером телефона уже существует (например, создан через станцию), backend не должен создавать дублирующую запись (см. открытые вопросы — точное поведение конфликта не зафиксировано).

#### Промоаккаунт

- Backend должен позволять выдавать и снимать статус промоаккаунта пользователю по запросу администратора с доступом к разделу «Пользователи».
- Точный эффект промоаккаунта на бизнес-логику (бесплатные аренды, скидка и т.п.) не зафиксирован во входных данных (см. открытые вопросы).

#### Блокировка

- Backend должен позволять блокировать пользователя по запросу администратора с доступом к разделу «Пользователи».
- Заблокированный пользователь не должен иметь возможности начать новую аренду через станцию (`sa/docs/kiosk/terminal-rental-flow`).
- Поведение для уже идущей (активной) аренды заблокированного пользователя не зафиксировано во входных данных (см. открытые вопросы).

#### Карточка пользователя

- Backend должен предоставлять данные карточки пользователя: имя, телефон, почту, текущую аренду (если есть), историю завершённых аренд.

#### Авторизация

- Backend должен проверять, что операции создания, выдачи/снятия промоаккаунта, блокировки и просмотра карточки пользователя выполняются администратором, которому назначен раздел «Пользователи», или супер-админом.

### Побочные эффекты и интеграционные результаты

- Платежи: N/A.
- Подписка / доступ: доступ к операциям регулируется набором разделов администратора (`sa/docs/admin/admin-roles`).
- Аналитика: Не определено.
- Email / CRM / webhooks: уведомляется ли пользователь о промоаккаунте или блокировке — не определено.

### Ограничения и исключения на BE

- Критерии не описывают разблокировку пользователя.
- Критерии не фиксируют точный эффект промоаккаунта.
- Критерии не описывают принудительное завершение активной аренды при блокировке.

## Нефункциональные критерии

### FE

- Производительность: Не определено.
- Доступность: раздел «Пользователи» не должен быть доступен прямым переходом по URL администратору без назначенного раздела.
- Консистентность UX: список и карточка должны явно отображать статусы промоаккаунта и блокировки.

### BE

- Надежность: Не определено.
- Наблюдаемость: Не определено.
- Безопасность: проверка доступа к разделу «Пользователи» должна выполняться на backend, а не только скрытием элементов на FE.

## Вне scope

- Редактирование данных пользователя (имя, телефон, почта) после создания.
- Удаление пользователя.
- Разблокировка пользователя.
- Уведомления пользователю о промоаккаунте/блокировке.
- Создание или изменение Qase test cases.

## Открытые вопросы

- Что именно даёт пользователю промоаккаунт?
- Что происходит при блокировке пользователя с активной арендой?
- Может ли заблокированный пользователь быть разблокирован, и если да — кем и как?
- Должен ли пользователь получать уведомление о выдаче промоаккаунта или блокировке?
- Какие поля обязательны при ручном создании пользователя администратором (email — не уточнено)?
- Что происходит при создании пользователя с номером телефона, уже существующим в системе?

## Примечания

- Критерии подготовлены на основе draft feature spec и уточнения SA по составу раздела «Пользователи».
- FE отвечает за список пользователей, форму создания, карточку пользователя и действия промоаккаунта/блокировки.
- Backend отвечает за хранение сущности пользователя (общей со станцией), статусы промоаккаунта/блокировки, историю аренд и проверку доступа по разделу.
