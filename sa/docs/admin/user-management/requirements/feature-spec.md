## Проблема

- У администратора нет отдельного раздела для управления пользователями: создания учётной записи вручную, выдачи промо-статуса, блокировки нарушителей и просмотра информации о пользователе и его арендах.
- Доступ к этому разделу должен быть управляем через ролевую модель `sa/docs/admin/admin-roles` (раздел «Пользователи» в наборе разделов администратора).

## Цель

- Дать администратору (с доступом к разделу «Пользователи») возможность видеть список пользователей, создавать пользователя, выдавать и снимать промоаккаунт, блокировать пользователя и просматривать карточку пользователя (имя, номер телефона, почта, текущая аренда при наличии, история аренд).

## Не входит в текущую итерацию

- Редактирование данных пользователя (имя, телефон, почта) после создания — не упомянуто во входных данных.
- Удаление пользователя.
- Разблокировка пользователя — не упомянута явно во входных данных (см. открытые вопросы).
- Точный эффект промоаккаунта (бесплатные аренды, скидка и т.п.) — не описан во входных данных.

## Триггер и аудитория

- Точка входа / триггер: администратор с доступом к разделу «Пользователи» открывает раздел в `gozap-admin`.
- Основная аудитория: администратор с назначенным разделом «Пользователи» (включая супер-админа).

## Целевое поведение продукта

### Сценарий

1. Администратор с доступом к разделу «Пользователи» открывает список пользователей в `gozap-admin`.
2. Администратор создаёт нового пользователя или открывает карточку существующего (созданного, в том числе, через станцию — см. `sa/docs/kiosk/terminal-rental-flow`).
3. Администратор может выдать или снять промоаккаунт пользователю, заблокировать пользователя, а также увидеть имя, номер телефона, почту, данные о текущей аренде (при наличии) и историю аренд пользователя.

### Правила результата

- Раздел «Пользователи» должен быть доступен только администратору, которому он назначен (или супер-админу) — см. `sa/docs/admin/admin-roles`.
- Список пользователей должен включать пользователей, созданных как вручную администратором, так и автоматически через станцию (`sa/docs/kiosk/terminal-rental-flow`).
- Карточка пользователя должна показывать: имя, номер телефона, почту, данные о текущей аренде (если есть активная аренда), историю аренд пользователя.
- Выдача/снятие промоаккаунта и блокировка должны быть доступны из карточки пользователя.

## Бизнес-правила

- Пользователь — это та же сущность «аккаунт», что создаётся автоматически через станцию по номеру телефона (`sa/docs/kiosk/terminal-rental-flow`); раздел «Пользователи» не создаёт отдельную, изолированную запись.
- Промоаккаунт и блокировка — это два независимых статуса пользователя, управляемых администратором.

## Пограничные сценарии

- Администратор блокирует пользователя с активной (незавершённой) арендой.
- Администратор создаёт пользователя с номером телефона, который уже существует в системе (создан через станцию).
- Заблокированный пользователь пытается арендовать powerbank через станцию.
- У пользователя нет ни одной завершённой аренды — карточка должна корректно отображать пустую историю.

## Открытые вопросы

- Что именно даёт пользователю промоаккаунт (бесплатные аренды, скидка, иное)?
- Что происходит при блокировке пользователя с активной арендой — аренда завершается принудительно или блокировка применяется только к будущим арендам?
- Может ли заблокированный пользователь быть разблокирован, и если да — кем и как?
- Должен ли пользователь получать уведомление (push/SMS) о выдаче промоаккаунта или блокировке?
- Какие поля обязательны при ручном создании пользователя администратором (телефон точно обязателен по аналогии со станцией; email — не уточнено)?

## Допущения

- Пользователь, создаваемый администратором вручную, использует ту же модель аккаунта, что создаётся автоматически через станцию (единая сущность пользователя в `gozap-backend`).
- Блокировка пользователя предполагается как минимум запретом новых аренд через станцию; точное поведение для уже идущей аренды не подтверждено.
