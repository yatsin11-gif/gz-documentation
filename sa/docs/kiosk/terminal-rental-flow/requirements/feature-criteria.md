# Критерии

## Источник

- Feature spec: `sa/docs/kiosk/terminal-rental-flow/requirements/feature-spec.md`
- Утвердил: Не утверждено; draft подготовлен по запросу SA
- Дата: 2026-06-30
- Репозитории в скоупе: `gozap-kiosk-app`, `gozap-backend`
- Связанные репозитории: `gozap-app` (получает синхронизированную аренду и push-уведомления; экраны приложения вне scope)

## Цель

- Пользователь должен иметь возможность арендовать powerbank на станции через ввод номера телефона, подтверждение SMS-кодом и привязку карты.
- Пользователь с разряженным телефоном должен иметь возможность арендовать powerbank только по карте, без ввода номера и регистрации.

## FE Criteria

### Затронутые поверхности

- `gozap-kiosk-app`: экран покоя, экран ввода номера телефона, экран ввода SMS-кода, экран привязки карты, экран упрощённого flow для разряженного телефона, экран успеха/ошибки.

### Пользовательские сценарии и состояния

#### Экран покоя

- Станция должна показывать основной интерфейс с полем ввода номера телефона.
- Станция должна показывать менее акцентную кнопку альтернативного сценария для разряженного телефона.

#### Ввод номера и подтверждение SMS

- Пользователь должен иметь возможность ввести номер телефона на экране станции.
- После ввода номера станция должна отправить запрос на подтверждение по SMS-коду.
- Пользователь должен иметь возможность ввести полученный код для подтверждения номера.
- После успешного подтверждения кода система должна создать аккаунт по номеру телефона.

#### Привязка карты и оплата

- После подтверждения номера станция должна показать экран привязки карты с инструкцией.
- Пользователь должен иметь возможность приложить карту для обработки и сохранения в аккаунте.
- После успешной оплаты станция должна выдать PowerBank пользователю.
- После успешной оплаты пользователь должен получить SMS о создании аккаунта/начале аренды.
- Если подтверждение SMS-кода не прошло, станция должна показать пользователю ошибку, не создавая аренду.
- Если оплата отклонена, станция должна показать пользователю ошибку, не создавая аренду и не выдавая PowerBank.

#### Flow для разряженного телефона

- Пользователь должен иметь возможность нажать отдельную кнопку для упрощённого flow на экране покоя.
- В упрощённом flow станция должна показать подсказку «Приложите карту для аренды».
- Пользователь должен иметь возможность оплатить картой без ввода номера телефона и без регистрации.
- После успешной оплаты в упрощённом flow станция должна выдать PowerBank.

### Синхронизация с приложением и уведомления

- Аренда, начатая через станцию, должна быть видна пользователю в `gozap-app`, если он авторизован тем же номером телефона.
- Пользователь должен получать push-уведомление (если `gozap-app` установлено) и SMS о начале аренды; то же правило действует для завершения аренды.
- Если у пользователя не установлено `gozap-app`, он должен получить только SMS, без push.

### Правила пользовательского поведения

- PowerBank не должен выдаваться без подтверждённой успешной оплаты ни в одном из двух flow.
- Основной flow обязательно проходит через подтверждение номера по SMS перед привязкой карты.
- Упрощённый flow не должен запрашивать номер телефона или создание аккаунта.
- Аренда должна создаваться только после подтверждённого номера телефона и успешной оплаты; при отказе на любом из этих шагов аренда не создаётся.

### Ограничения и исключения на FE

- Критерии не фиксируют точный визуальный дизайн экранов станции.
- Критерии не описывают экран/правила окончания аренды и возврата powerbank (вне текущих входных данных).
- Критерии не требуют альтернативных способов оплаты, кроме банковской карты.

## BE Criteria

### Затронутые backend области

- `gozap-backend`: создание аккаунта пользователя станции, отправка и проверка SMS-кода, обработка оплаты картой, создание сущности «аренда», выдача PowerBank, синхронизация аренды с `gozap-app`, отправка уведомлений (push + SMS) о начале и завершении аренды.
- Внешние интеграции: Twilio (SMS-код подтверждения и уведомления); push-провайдер `gozap-app` не подтверждён; платёжный провайдер для приёма карты на терминале не подтверждён.

### Правила бизнес-логики и обработки

#### Подтверждение номера и создание аккаунта

- Backend должен принимать номер телефона от станции и инициировать отправку SMS-кода через Twilio.
- Backend должен проверять введённый SMS-код перед созданием аккаунта.
- При успешной проверке кода backend должен создать аккаунт пользователя, привязанный к номеру телефона.

#### Оплата, создание аренды и выдача PowerBank

- Backend должен принимать данные карты для обработки оплаты и сохранять карту в аккаунте пользователя для будущих списаний.
- Backend должен создавать сущность «аренда» только после подтверждённого номера телефона и успешной оплаты.
- Backend должен выдавать команду станции на выдачу PowerBank только после подтверждения успешной оплаты и создания аренды.
- Если оплата отклонена, backend не должен создавать аренду и не должен выдавать команду на PowerBank.

#### Синхронизация с приложением и уведомления

- Аренда, созданная через станцию, должна быть той же сущностью, что доступна через `gozap-app` (не отдельная изолированная запись).
- Backend должен синхронизировать аренду в `gozap-app`, если пользователь авторизован в приложении тем же номером телефона.
- Backend должен отправлять push-уведомление (если `gozap-app` установлено) и SMS через Twilio о начале аренды; то же правило действует для завершения аренды.
- Если номер телефона станции не совпадает с аккаунтом в `gozap-app`, аренда не должна синхронизироваться в чужой аккаунт.

#### Упрощённый flow для разряженного телефона

- Backend должен поддерживать оплату картой без номера телефона и без создания полноценного аккаунта.
- Backend должен выдавать команду станции на выдачу PowerBank в упрощённом flow только после подтверждения успешной оплаты.

### Побочные эффекты и интеграционные результаты

- Платежи: точный платёжный провайдер для обработки карты на терминале не подтверждён (отдельно от Stripe для партнёрских выплат).
- Подписка / доступ: N/A.
- Аналитика: Не определено.
- Email / CRM / webhooks: SMS-уведомления через Twilio на этапах подтверждения номера, начала и завершения аренды; push-уведомления через `gozap-app` (провайдер не подтверждён) на начало и завершение аренды.

### Ограничения и исключения на BE

- Критерии не фиксируют flow окончания аренды, возврата powerbank и тарификации по времени.
- Критерии не требуют поддержки нескольких способов оплаты.
- Критерии не фиксируют точный платёжный провайдер для терминала.

## Нефункциональные критерии

### FE

- Производительность: пользователь должен видеть явные состояния ожидания на этапах отправки SMS, проверки кода и обработки оплаты.
- Доступность: подсказки на экране должны быть понятны без предварительного обучения (использование станции разовое).
- Консистентность UX: оба flow (основной и упрощённый) должны заканчиваться одинаковым экраном успеха/выдачи PowerBank.

### BE

- Надежность: backend должен возвращать понятную ошибку, если SMS-код не подтверждён или оплата отклонена, без выдачи PowerBank.
- Наблюдаемость: backend должен различать ошибки отправки SMS, ошибки проверки кода и ошибки оплаты.
- Безопасность: backend не должен сохранять полные данные карты напрямую — обработка карты должна идти через платёжного провайдера.

## Вне scope

- Окончание аренды и возврат powerbank.
- Альтернативные способы оплаты, кроме карты.
- Создание или изменение Qase test cases.

## Примечания

- Критерии подготовлены на основе draft feature spec и входных SA-заметок по terminal rental flow.
- FE отвечает за экраны станции: покой, ввод номера, ввод кода, привязка карты, упрощённый flow, успех/ошибка.
- Backend отвечает за подтверждение номера через Twilio, создание аккаунта, обработку оплаты и команду выдачи PowerBank.
