## Проблема

- Пользователю у станции нужен быстрый способ арендовать powerbank без скачивания приложения.
- Пользователь с разряженным телефоном не может пройти полный flow с вводом номера и SMS-подтверждением.

## Цель

- Дать пользователю возможность арендовать powerbank на экране станции через ввод номера телефона, подтверждение по SMS и привязку карты.
- Дать пользователю с разряженным телефоном упрощённый flow аренды только по карте, без ввода номера и регистрации.

## Не входит в текущую итерацию

- Аренда через мобильное приложение или QR-сканирование сторонним приложением.
- Возврат powerbank и расчёт стоимости за фактическое время аренды (флоу окончания аренды не описан во входных данных).
- Множественные способы оплаты, кроме банковской карты.
- Программы лояльности или промокоды.

## Триггер и аудитория

- Точка входа / триггер: пользователь подходит к станции, которая находится в режиме экрана покоя.
- Основная аудитория: пользователь с рабочим телефоном, готовый подтвердить номер по SMS.
- Вторичная аудитория: пользователь с разряженным телефоном, которому нужен flow без ввода номера.

## Целевое поведение продукта

### Сценарий

1. Пользователь подходит к станции и видит экран покоя с полем ввода номера телефона и менее акцентной кнопкой альтернативного сценария (для разряженного телефона).
2. Пользователь вводит номер телефона и подтверждает его кодом из SMS; система создаёт аккаунт по номеру телефона. Если код не подтверждён, пользователь видит ошибку на станции и аренда не начинается.
3. Пользователь прикладывает карту, она обрабатывается и сохраняется в аккаунте для будущих списаний. Если оплата отклонена, пользователь видит ошибку на станции и аренда не начинается.
4. После успешной оплаты пользователь получает PowerBank, аренда создаётся и становится видна в мобильном приложении `gozap-app` (если оно установлено и пользователь авторизован тем же номером), а пользователь получает push-уведомление (если приложение установлено) и SMS о начале аренды. Та же логика уведомлений (push + SMS) действует при завершении аренды.

### Правила результата

- Аккаунт должен создаваться автоматически по номеру телефона после подтверждения SMS-кода.
- Карта должна сохраняться в аккаунте пользователя для будущих списаний.
- Аренда должна создаваться только после подтверждённого номера телефона и успешной оплаты; PowerBank должен выдаваться только после успешной оплаты.
- Если подтверждение номера по SMS не прошло, пользователь должен увидеть ошибку на экране станции, и аренда не должна создаваться.
- Если оплата отклонена, пользователь должен увидеть ошибку на экране станции, аренда не должна создаваться и PowerBank не должен выдаваться.
- Аренда, начатая через станцию (терминал), должна быть видна пользователю в `gozap-app`, если он авторизован тем же номером телефона.
- Пользователь должен получать push-уведомление (если приложение установлено) и SMS о начале аренды; то же правило действует для момента завершения аренды.
- В сценарии разряженного телефона PowerBank должен выдаваться без ввода номера телефона и без создания аккаунта.

## Бизнес-правила

- Основной flow требует подтверждения номера телефона по SMS перед привязкой карты.
- Flow для разряженного телефона запускается отдельной, менее акцентной кнопкой на экране покоя.
- В сценарии разряженного телефона оплата проходит сразу по карте, без ввода номера и регистрации.
- SMS-код для подтверждения номера отправляется через Twilio.
- Аренда, начатая через станцию, — это та же сущность «аренда», что видна пользователю в `gozap-app`; терминал не создаёт отдельную, изолированную от приложения запись.
- Уведомление о начале и о завершении аренды отправляется по двум каналам одновременно: push (если приложение установлено) и SMS (всегда, независимо от наличия приложения).

## Ограничения и известные лимитации

- Продукт / UX: основной экран должен явно выделять поле ввода номера и менее акцентно — альтернативный сценарий для разряженного телефона.
- Биллинг / правовые ограничения: Не определено — точный платёжный провайдер для приёма карты на терминале во входных данных не назван (отдельно от Stripe, который подтверждён только для партнёрских выплат).
- Технические ограничения, уже подтвержденные на продуктовом уровне: подтверждение номера телефона выполняется кодом из SMS через Twilio.
- Внешние зависимости: Twilio (отправка SMS с кодом подтверждения и уведомлением о начале аренды).

## Влияние на аналитику, CRM и коммуникации

- Аналитика: Не определено.
- CRM / подписочные системы: Не определено.
- Письма / уведомления: пользователь получает push (если установлено `gozap-app`) и SMS (Twilio) о начале и о завершении аренды, начатой через станцию.

## Пограничные сценарии

- Пользователь не получает SMS-код вовремя; аренда не должна начинаться.
- Карта пользователя не проходит обработку (отказ оплаты); аренда не должна начинаться.
- Пользователь нажимает кнопку «разряженный телефон», но затем хочет вернуться к полному flow.
- Оплата прошла, но станция не выдала PowerBank.
- У пользователя нет установленного `gozap-app` — должен прийти только SMS, без push.
- Пользователь авторизован на станции номером телефона, который не совпадает с аккаунтом в `gozap-app`, — аренда не должна синхронизироваться в чужой аккаунт.

## Допущения

- «Инструкция по карте» и «код из смс» из исходных заметок — это подсказки/состояния экрана в рамках одного и того же flow аренды, а не отдельные фичи.
- Экран окончания аренды относится к этому же продукту, но его бизнес-правила не подтверждены и не входят в эту итерацию спецификации; правило уведомлений на завершение аренды фиксируется заранее, чтобы будущая итерация его учитывала.
- `gozap-app` уже существует как отдельный продукт пользователя; эта фича добавляет в него только синхронизацию аренды и приём push, без проектирования экранов приложения.
