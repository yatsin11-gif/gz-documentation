## Problem Statement

Сейчас нет backend-модели партнёра, привязки станций к партнёру с индивидуальным процентом вознаграждения и расчёта начислений по периоду фактической привязки. Без этого фундамента невозможно построить статистику для администратора и партнёра.

## Scope

- CRUD учётной записи партнёра (логин, пароль, имя, телефон), включая редактирование данных уже созданного партнёра
- список партнёров с поиском по логину (частичное совпадение) и сортировкой для администратора (по умолчанию по дате добавления от новых к старым; также по логину и по дате добавления по возрастанию/убыванию)
- привязка/отвязка одной или нескольких станций к партнёру
- хранение процента вознаграждения отдельно для каждой пары «станция-партнёр», включая 0%
- гарантия, что станция привязана не более чем к одному партнёру одновременно
- начисление доли партнёра в баланс при завершении аренды по проценту, действовавшему на момент завершения; процент применяется к базе — реально полученному доходу станции по аренде после обязательных вычетов по платежам (комиссий Stripe), не к полному обороту аренды и не к чистой прибыли
- учёт начислений только за период фактической привязки станции к партнёру, без пересчёта прошлого при отвязке/повторной привязке
- эндпоинты статистики: список станций партнёра, сводка за период, детализация аренд по станции

## Out of Scope

- интеграция со Stripe для выплат
- сброс пароля через SMS
- frontend-интерфейсы

## Implementation Notes

- affected modules: модуль партнёров/станций, модуль аренд/начислений backend (точные пути модулей `gozap-backend` требуют подтверждения при старте реализации)
- constraints: расчёт начисления должен быть идемпотентным относительно повторной обработки одной аренды; база начисления — реально полученный доход станции по аренде после обязательных вычетов по платежам (комиссий Stripe); нужен доступ к сумме комиссий Stripe по каждой аренде
- known technical context: точная текущая схема станций и аренд в `gozap-backend` не исследована в рамках этого документа — подтвердить перед реализацией

## Acceptance Criteria

1. Администратор может создать партнёра с логином, паролем, именем и телефоном.
2. Администратор может отредактировать данные уже созданного партнёра (логин, имя, телефон).
3. Администратор может искать партнёров по логину (частичное совпадение) и сортировать список по логину и по дате добавления (по умолчанию — по дате добавления от новых к старым).
4. Администратор может привязать/отвязать одну или несколько станций к партнёру с процентом вознаграждения (включая 0%).
5. При завершении аренды backend начисляет долю партнёра в баланс по проценту, действовавшему на станции в момент завершения, применяя процент к реально полученному доходу станции по аренде после обязательных вычетов по платежам (комиссий Stripe), не к полному обороту аренды и не к чистой прибыли; при последующем изменении процента ранее начисленные доли не пересчитываются.
6. Баланс партнёра учитывает только аренды, завершённые в период фактической привязки станции: после отвязки аренды, завершённые позже, не учитываются; после повторной привязки аренды, завершённые позже, снова учитываются.
7. API отдаёт данные для списка станций партнёра, сводки за период и детализации по станции.
