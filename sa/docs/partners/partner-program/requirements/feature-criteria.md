# Критерии

## Источник

- Feature spec: `sa/docs/partners/partner-program/requirements/feature-spec.md`
- Утвердил: Не утверждено; draft подготовлен по запросу SA
- Дата: 2026-06-30
- Репозитории в скоупе: `gozap-admin`, `gozap-backend`

## Цель

- Администратор должен иметь возможность завести партнёра, привязать к нему станции с индивидуальным процентом вознаграждения и видеть ту же статистику, что видит партнёр.
- Партнёр должен видеть только свои данные: станции, аренды, баланс, выплаты — и получать вознаграждение за предыдущий месяц автоматически 7-го числа каждого месяца через Stripe.

## FE Criteria

### Затронутые поверхности

- `gozap-admin`: раздел управления партнёрами (admin-роль), личный кабинет партнёра (partner-роль), логин и сброс пароля.
- `gozap-admin` (партнёр): список станций, сводка за период, детализация по станции, кнопка перехода в Stripe для указания реквизитов выплат, кнопка перехода в Stripe для просмотра истории начислений.

### Пользовательские сценарии и состояния

#### Создание и управление партнёром (администратор)

- Администратор должен иметь возможность создать партнёра, указав логин, пароль, имя и телефон.
- Администратор должен иметь возможность отредактировать данные уже созданного партнёра (логин, имя, телефон).
- Администратор должен иметь возможность искать партнёров в списке по логину с частичным совпадением.
- Список партнёров по умолчанию должен сортироваться по дате добавления от новых к старым; администратор должен иметь возможность переключить сортировку на логин (по возрастанию/убыванию) или на дату добавления (по возрастанию/убыванию).
- Администратор должен иметь возможность привязать одну или несколько станций к партнёру и отвязать их.
- Администратор должен иметь возможность задать процент вознаграждения отдельно для каждой привязанной станции, включая значение 0%.
- Администратор должен иметь возможность открыть представление, идентичное тому, что видит партнёр, для любой выбранной точки/партнёра.

#### Вход и восстановление доступа (партнёр)

- Партнёр должен иметь возможность залогиниться и попасть в раздел, где видны только его данные.
- Партнёр должен иметь возможность запросить сброс пароля через SMS на свой номер телефона.

#### Список станций партнёра

- Партнёр должен видеть список своих станций: код, название, адрес точки, статус.
- Для каждой станции партнёр должен видеть количество аренд и сумму вознаграждения за выбранный период.

#### Сводка за период

- Партнёр должен видеть текущий баланс.
- Партнёр должен видеть размер ближайшей выплаты.
- Партнёр должен видеть число завершённых аренд за период.
- Партнёр должен видеть сумму вознаграждения за период.
- Партнёр должен видеть разбивку вознаграждения по станциям за период.

#### Детализация по станции

- Партнёр должен иметь возможность открыть конкретную станцию и увидеть список аренд за период: дата, стоимость, вознаграждение партнёра.
- Партнёр должен видеть итоговую сумму по списку аренд станции за период.

#### Выплаты

- Партнёр должен иметь возможность нажать кнопку и перейти на сторону Stripe, чтобы зарегистрироваться или авторизоваться и указать там данные банковского счёта/карты; форма ввода/редактирования карты в самом `gozap-admin` не предусмотрена.
- Партнёр должен иметь возможность нажать кнопку и перейти (редирект) на сторону Stripe, чтобы посмотреть историю начислений; в своём профиле в Stripe партнёр также видит историю выплат.

### Правила пользовательского поведения

- Партнёр не должен иметь доступа к данным других партнёров или станций, к нему не привязанных.
- Администратор должен иметь возможность переключиться на представление любого партнёра без дополнительной авторизации от имени партнёра.
- Партнёр не должен иметь интерфейса для ручного запроса выплаты — выплата только автоматическая.

### Ограничения и исключения на FE

- Критерии не фиксируют точный визуальный дизайн экранов админки и кабинета партнёра.
- Критерии не требуют поддержки нескольких банковских счетов на одного партнёра.
- Критерии не требуют email-уведомлений партнёру (открытый вопрос в feature spec).

## BE Criteria

### Затронутые backend области

- `gozap-backend`: учётные записи партнёров и администраторов, привязка станций к партнёрам, расчёт баланса по аренде, начисления, выплаты.
- Внешние интеграции: Stripe (выплаты партнёрам), Twilio (SMS для сброса пароля).

### Правила бизнес-логики и обработки

#### Создание партнёра и привязка станций

- Backend должен поддерживать создание учётной записи партнёра с логином, паролем (захешированным), именем и телефоном.
- Backend должен поддерживать редактирование данных уже созданного партнёра (логин, имя, телефон).
- Backend должен поддерживать привязку и отвязку одной или нескольких станций к партнёру.
- Backend должен хранить процент вознаграждения отдельно для каждой пары «станция-партнёр», включая 0%.
- Одна станция должна быть привязана не более чем к одному партнёру одновременно.
- Backend должен поддерживать поиск партнёров по логину с частичным совпадением.
- Backend должен поддерживать сортировку списка партнёров по логину (по возрастанию/убыванию) и по дате добавления (по возрастанию/убыванию), с сортировкой по умолчанию по дате добавления от новых к старым.

#### Начисление баланса

- При завершении аренды backend должен рассчитывать долю партнёра по проценту, действовавшему на станции в момент завершения аренды, применяя процент к реально полученному доходу станции по аренде после обязательных вычетов по платежам (не к полному обороту аренды и не к чистой прибыли), и добавлять полученную долю в баланс партнёра.
- Backend должен определять базу начисления как доход станции по аренде после обязательных платёжных вычетов — комиссий Stripe; полный оборот аренды и чистая прибыль базой не являются.
- При изменении процента вознаграждения на станции backend должен применять новый процент только к арендам, завершённым после момента изменения; ранее начисленные доли не пересчитываются.
- Backend должен учитывать аренды в балансе партнёра только за период, когда станция была к нему привязана: после отвязки станции аренды, завершённые позже, не должны учитываться в балансе; после повторной привязки аренды, завершённые позже, должны снова учитываться.
- Backend должен хранить историю начислений с привязкой к станции, аренде и периоду, включая базу начисления (доход станции после обязательных вычетов) и применённый процент, для последующей сверки.

#### Выплаты через Stripe

- Backend должен формировать ссылку (Stripe-редирект) для регистрации/авторизации партнёра на стороне Stripe и указания им банковского счёта/карты; backend не должен сохранять данные карты партнёра — хранится только идентификатор соответствующего аккаунта/клиента Stripe.
- Backend должен формировать ссылку (Stripe-редирект) для просмотра партнёром истории начислений и истории выплат в его профиле в Stripe.
- Backend должен 7-го числа каждого месяца автоматически инициировать выплату суммы вознаграждения за предыдущий месяц через Stripe.
- После успешной выплаты backend должен обнулять выплаченную часть баланса и сохранять запись в истории выплат (сумма, дата, статус).
- Если выплата через Stripe завершается ошибкой, backend должен сохранить статус ошибки и не терять накопленный баланс партнёра.
- Если на момент плановой выплаты 7-го числа у партнёра не указан банковский счёт/карта, backend должен отложить выплату накопленного баланса без потери средств; как только партнёр указывает счёт/карту, backend должен инициировать выплату накопленного баланса немедленно, после чего выплаты продолжаются по расписанию (7-го числа каждого месяца).

#### Сброс пароля и доступ

- Backend должен поддерживать сброс пароля партнёра через SMS-код на номер телефона (Twilio).
- Backend должен ограничивать доступ партнёра к API только данными станций, привязанных к нему.

### Побочные эффекты и интеграционные результаты

- Платежи: автоматическая ежемесячная выплата через Stripe; реквизиты счёта/карты партнёр вводит на стороне Stripe через hosted-редирект (регистрация/авторизация в Stripe), что указывает на Connect-подобную модель; точный продукт линейки Stripe Connect не подтверждён.
- Подписка / доступ: N/A.
- Аналитика: Не определено.
- Email / CRM / webhooks: backend должен отправлять SMS через Twilio при сбросе пароля; webhook от Stripe о статусе выплаты не подтверждён как обязательный.

### Ограничения и исключения на BE

- Критерии не фиксируют точную схему БД, имена таблиц или внутренние идентификаторы.
- Критерии не требуют поддержки нескольких валют.
- Критерии не требуют ручного триггера выплаты со стороны партнёра или администратора.

## Нефункциональные критерии

### FE

- Производительность: список станций, сводка и детализация по станции должны явно показывать состояние загрузки.
- Доступность: разделы кабинета партнёра должны быть недоступны без аутентификации.
- Консистентность UX: представление администратора «от лица партнёра» должно использовать тот же UI, что и кабинет партнёра.

### BE

- Надежность: расчёт баланса по завершённой аренде должен быть идемпотентным — повторная обработка одной и той же аренды не должна задваивать начисление.
- Наблюдаемость: backend должен различать ошибки расчёта начисления и ошибки выплаты через Stripe.
- Безопасность: доступ партнёра к API должен быть строго ограничен его собственными станциями и данными.

## Вне scope

- Ручной запрос выплаты партнёром.
- Несколько банковских счетов на одного партнёра.
- Поддержка нескольких валют.
- Собственная форма ввода/хранения данных банковского счёта/карты партнёра в `gozap-admin`/`gozap-backend` — данные карты вводятся и хранятся только на стороне Stripe.
- Собственная страница истории начислений/выплат в `gozap-admin` — оба раздела доступны партнёру через редирект в Stripe.
- Создание или изменение Qase test cases.

## Открытые вопросы

- Какой именно продукт линейки Stripe Connect (Standard/Express/Custom) используется и какой объём верификации (KYC) Stripe требует от партнёра при регистрации?
- Нужны ли уведомления партнёру о начислении и о выплате, кроме SMS при сбросе пароля?

## Примечания

- Критерии подготовлены на основе draft feature spec и входных SA-заметок по партнёрской программе.
- FE отвечает за разделы администратора и партнёра в `gozap-admin`, включая представление «как у партнёра».
- Backend отвечает за привязку станций, расчёт и хранение баланса по периодам привязки, интеграцию со Stripe и Twilio.
